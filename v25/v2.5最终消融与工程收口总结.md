# v2.5 最终消融与工程收口总结

更新时间：2026-06-28

这份文档用于固化 v2.5 阶段的最终工程结论。重点不是继续追求单一算法的跑分，而是明确当前系统中每一层的真实作用：A* 负责可预测层全局规划，APAS 负责不可预测层下的硬安全兜底，waypoint-skip 修复路径跟踪进度问题，Expert 负责降低风险暴露和安全盾介入负担。

## 1. 当前系统主线

v2.5 的核心结构可以概括为：

```text
Predictable Layer
    -> A* global planning
    -> reference path / waypoint tracking

Random True-World Layer
    -> local wind disturbance
    -> storm risk / destructive core
    -> true execution uncertainty

Execution Stack
    -> A* reference command
    -> optional Expert residual decision
    -> optional APAS safety filter
    -> true-world dynamics
    -> metrics / failure trace / ablation
```

当前最稳定的工程闭环是：

```text
A* + APAS + waypoint-skip
```

在这个闭环上再加入 Expert 后，成功率不一定继续提高，但安全介入负担和风险暴露会明显下降。

## 2. 各模块定位

### 2.1 A*

A* 只负责可预测层上的全局路径规划。它不主动读取随机层，也不会根据局部风险膜进行绕行决策。

在执行阶段，如果没有 APAS 或 Expert，A* baseline 的动作可以理解为：

```text
沿 A* 参考路线飞行，residual action = 0
```

因此裸 A* 在随机层扰动下容易失败，主要失败类型包括：

- storm_risk_too_high
- destructive_storm_core
- timeout
- terrain_or_nfz

### 2.2 waypoint-skip

waypoint-skip 是路径跟踪修复，不是避障算法。

它解决的问题是：无人机在风扰、APAS 修正或局部绕行后，可能错过某个 waypoint，但传统逻辑仍要求它进入旧 waypoint 的刷新半径，导致系统追逐已经过期的路径点，最终 timeout。

waypoint-skip 的核心逻辑是：

```text
如果当前位置最近的路径点已经明显超过 current_wp_idx，
并且飞机仍在路径走廊附近，
则允许 current_wp_idx 跳到更靠前的 waypoint。
```

它对 A*、A*+APAS、A*+Expert 都生效，因此它会提高所有方法的路径跟踪稳定性。

### 2.3 APAS

APAS 是单步预测式安全过滤器，不是全图规划器，也不是 Expert/MPC。

它的输入是上层期望动作：

```text
desired_heading_deg
desired_airspeed_mps
desired_agl_m
```

它围绕期望动作生成候选动作：

```text
heading offsets: 0, ±15, ±30, ±45, ±60, ±90 deg
speed decrement: 每次降低 2 m/s
AGL increment: 0, 30, 60 m
```

每个候选动作会经过：

```text
1. true-world one-step transition
2. endpoint safety check
3. swept-segment safety check
4. first-safe candidate selection
5. least-bad fallback if no candidate is fully safe
```

APAS 不看全图。它只检查当前步候选运动段，即：

```text
current_pos -> candidate_next_pos
```

但 APAS 查询的是当前局部真实随机层，所以它是一个强局部安全盾。

### 2.4 Expert

Expert 是局部风险决策器，不是全局重规划器。它的作用不是替代 A*，而是在 A* 参考动作基础上给出残差修正。

当前 Expert 包含：

```text
local risk observer
risk membrane / band avoidance
short-horizon candidate rollout
normal / cautious / avoiding / emergency / recovering modes
do-no-harm gate
```

最终实验显示，Expert 的主要价值不是显著提高成功率，而是：

```text
降低平均风险
降低峰值风险
减少 APAS 强制接管
减少 APAS segment rejection
降低 adjusted mission cost
```

因此 Expert 更准确的定位是：

```text
safety burden reducer / upper-level risk-aware local controller
```

而不是：

```text
success-rate booster
```

## 4. 消融结论

### 4.1 waypoint-skip 的贡献

```text
A* raw:             40.0%
A* + waypoint-skip: 50.0%
```

waypoint-skip 单独带来约 +10% 成功率提升。

失败原因变化：

```text
timeout: 6 -> 2
goal_reached: 12 -> 15
```

这说明 timeout 中确实存在一类由 waypoint 进度卡死导致的问题。该问题不是策略不聪明，而是路径跟踪器在扰动执行后仍追逐旧 waypoint。

### 4.2 APAS 的贡献

```text
A* raw:     40.0%
A* + APAS: 53.3%
```

APAS 单独带来约 +13.3% 成功率提升。

失败原因变化：

```text
storm_risk_too_high: 8 -> 2
goal_reached:        12 -> 16
```

这说明 APAS 对随机层风险有明显兜底作用。但 APAS 单独使用时也会增加任务时间、路径距离、能耗和安全干预负担：

```text
APAS interventions mean:      11.33
segment rejections mean:      78.27
safety burden mean:           135.65
adjusted energy mean:         890203 J
```

### 4.3 APAS + waypoint-skip 的组合贡献

```text
A* raw:                    40.0%
A* + APAS + waypoint-skip: 76.7%
```

这是当前系统成功率提升的主力组合，带来约 +36.7% 的提升。

该组合说明：

```text
APAS 解决随机层局部危险兜底；
waypoint-skip 解决安全修正后的路径进度恢复；
二者组合形成稳定执行闭环。
```

这也是 v2.5 当前最重要的工程结论。

### 4.4 Expert 的贡献

成功率：

```text
A* + APAS + waypoint-skip:            76.7%
A* + Expert + APAS + waypoint-skip:   76.7%
```

Expert 没有在这组 30 episodes 中进一步提高成功率。

但 Expert 显著降低了风险和安全干预负担：

```text
avg_risk:                 0.0699 -> 0.0476
peak_risk:                0.3072 -> 0.2378
APAS interventions:       11.90 -> 1.60
APAS segment rejections:  86.73 -> 1.67
adjusted energy:          788099 J -> 712658 J
```

相对变化：

```text
avg_risk 下降约 32%
peak_risk 下降约 23%
APAS interventions 下降约 87%
segment rejections 下降约 98%
adjusted mission cost 下降约 9.6%
```

因此 Expert 的真实价值是：

```text
在不降低成功率的情况下，让系统更少依赖 APAS 硬接管，并降低风险暴露和综合任务代价。
```

这比单纯宣称“Expert 提高成功率”更准确，也更适合作为工程系统贡献。

## 5. 最终叙事

当前 v2.5 最稳的表述是：

```text
A* provides interpretable global planning on the predictable layer.
APAS provides a one-step predictive safety filter for local random-layer hazards.
Waypoint-skip repairs path-progress loss caused by wind, avoidance, and safety interventions.
The Expert controller does not primarily improve raw success rate under a strong APAS shield; instead, it reduces risk exposure and safety intervention burden.
```

中文表述：

```text
A* 负责可预测层上的可解释全局路径规划；
APAS 负责不可预测随机层下的单步硬安全兜底；
waypoint-skip 修复扰动执行和安全接管后出现的路径进度丢失；
Expert 在强 APAS 兜底下并不主要提升原始成功率，而是降低风险暴露和安全干预负担。
```

更口语化的系统分工：

```text
A* 负责知道往哪走；
waypoint-skip 负责别追过期路点；
APAS 负责别死；
Expert 负责别老是等 APAS 来救。
```

## 6. 为什么这可以算工程收口

v2.5 目前已经具备：

```text
1. 分层地图建模
2. 可预测层 A* 全局规划
3. 随机层 true-world 执行
4. 局部风险观测
5. APAS 单步安全过滤
6. waypoint 进度修复
7. Expert 局部风险决策
8. failure trace / hazard classification
9. adjusted energy / safety intervention burden
10. final ablation
```

这意味着项目已经从“算法拼接 demo”进入了“可解释、可诊断、可消融的工程系统”阶段。

当前不建议继续无边界调参。更合适的后续工作是：

```text
1. 整理命名，避免把 A* + APAS 误写成 A* only。
2. 保留 final ablation 作为 v2.5 收口实验。
3. 将 Expert 定位为 safety burden reducer，而非成功率主贡献。
4. 如果继续研究，再单独开 v2.6 分支处理 RL imitation / multi-seed holdout / richer local planner。
```

## 7. 后续可选方向

如果未来继续推进，可以考虑：

```text
1. 换 seed 区间做 holdout 验证，例如 seed=200, episodes=30。
2. 让 RL 学 Expert，而不是直接从稀疏任务奖励学避障。
3. 将 APAS 从 one-step safety filter 扩展为更正式的 predictive safety filter。
4. 将 Expert 的有限候选动作升级为更规范的局部轨迹库。
5. 对 risk membrane 做采样密度/角度分辨率消融。
6. 针对 destructive core 的 sensor-blind case 单独分析感知边界。
```

但这些都属于下一阶段研究，而不是 v2.5 工程收口必须完成的内容。

## 8. 当前结论

v2.5 可以阶段性收口。

最终结论不是“RL/Expert 神奇地超过 A*”，而是：

```text
在分层地图和随机执行场景中，单纯 A* 不足以稳定完成任务；
APAS 与 waypoint-skip 构成主要成功率提升来源；
Expert 在强安全盾存在时主要降低风险暴露和安全干预负担；
系统整体已经形成可解释的分层安全规划与执行闭环。
```

## 9. 2026-06-29 至 2026-07-01 收尾更新过程

```text
1. Expert 已经能够降低风险与 APAS 负担，RL 是否可以学习/压缩 Expert？
```

### 9.3 PPO Fine-tune 与 seed 43 病理

在 BC 模型基础上进行了 50k PPO fine-tune：

```powershell
python v25\train_rl_disruptive.py --run-name ppo_v25_expert_bc_s200_ft50k --seed 200 --curriculum-stage 3 --sensor-mode circle_oracle --stress fragile --enable-apas --total-timesteps 50000 --n-envs 1 --model-save-root v25\artifacts\models\ppo_v25_expert_bc_s200_ft50k --load-model-path v25\artifacts\models\ppo_v25_expert_bc_s200 --log-root v25\artifacts\logs
```

seed=200 的 30 episodes 对比中，FT-PPO 有一组不错的结果：

```text
A*:      success 83.3%
Expert:  success 83.3%
FT-PPO:  success 86.7%

FT-PPO rescued: 1
FT-PPO harmed:  0
```

但 seed=42 的小样本 smoke test 暴露了 seed 43 病理：

```text
A*+APAS:      seed 43 success
Expert+APAS:  seed 43 fail
FT-PPO+APAS:  seed 43 fail
BC-PPO+APAS:  seed 43 success, but with very high APAS burden
```

这说明问题不是 APAS 不够强，而是：

```text
上层策略在某些场景里主动干预后，反而把 A*+APAS 本来能通过的 case 拖进失败。
```

seed 43 的具体病理是：

```text
Expert 看到风险墙后进入 pre_emergency_slow，
但没有有效横向逃逸，
导致飞机在风险墙前减速、暴露时间增加、p_crash 持续上升，
最终 storm_risk_too_high。

FT-PPO 则出现负进度和路径漂移，
APAS 持续接管但仍然进入 no-valid / high-risk 状态。
```

因此这一阶段的关键收获是：

```text
Expert / PPO 不仅要“能避险”，还必须“不能伤害 A*+APAS 已经能完成的 case”。
```

### 9.4 Do-no-harm Rollback Gate

为了解决 seed 43 这类 harmed case，新增了 do-no-harm rollback gate。

代码位置：

```text
v25/rl_env_disruptive.py
configs/config.py
```

执行链路变为：

```text
A* base command
    -> Expert / PPO residual proposal
    -> do-no-harm rollback gate
    -> APAS final safety shield
    -> true-world dynamics
```

APAS 负责判断单步动作是否安全；do-no-harm gate 负责判断上层 residual 是否正在帮倒忙。

它只使用已经发生的历史信息，不读取未来随机层：

```text
recent progress
recent p_crash change
recent APAS interventions
recent APAS segment rejections
recent APAS no-valid candidates
local risk-wall / no-escape-gap indicators
```

当前包含两类触发：

```text
bad_progress_risk_not_improving:
    residual 已经活跃，但最近几步进度不佳，风险没有下降，APAS 也已经开始吃力。

slow_into_risk_wall:
    上层策略正在强减速，前方存在风险墙/无逃逸缝，风险没有下降。
    这专门对应 seed 43 的“减速泡在危险里”病理。
```

触发后：

```text
residual action -> 0
fallback to A* base command
APAS still performs final safety filtering
```

同时新增指标：

```text
episode_do_no_harm_events
episode_do_no_harm_suppressed_steps
episode_do_no_harm_cooldown_steps
```

这些指标已经接入：

```text
compare_astar_rl_disruptive.py
diagnose_failure_traces.py
final_ablation_v25.py
train_rl_disruptive.py
```

### 9.5 Do-no-harm 验证结果

seed 43 病理复测：

```text
A*+APAS:      success
Expert+APAS:  success
FT-PPO+APAS:  success
```

这说明 do-no-harm gate 能修复典型 harmed case。

但需要注意：Expert 被救回 seed 43 时，gate 触发很多，APAS 负担也很高。因此 do-no-harm gate 不是性能优化器，而是保险丝：

```text
它的价值是避免上层策略把可完成任务拖死；
不是让 Expert/PPO 在所有场景中都变得更优。
```

targeted 7 seeds 结果：

```text
A*+APAS:      57.1%
Expert+APAS:  71.4%
RL+APAS:      71.4%
```

这说明在专门挑选的病理种子集合中，rollback gate 有明确价值。

### 9.7 收尾后的最终定位

最终不建议把本阶段结果包装成：

```text
RL 显著提高成功率。
```

更准确的表述是：

```text
在强 APAS 安全盾下，A*+APAS+waypoint-skip 构成高成功率基线；
Expert 提供可解释的局部风险决策，并在部分病理场景中提高成功率；
PPO 学生策略可以压缩 Expert 的部分行为，在不显著降低成功率的情况下减少 APAS 介入与 adjusted mission cost；
do-no-harm gate 则用于防止 Expert/PPO 在少数场景中伤害 A*+APAS 基线。
```

最终系统叙事可以固定为：

```text
A*:
    可解释全局规划。

waypoint-skip:
    修复风扰 / APAS / 避障后路径进度丢失。

APAS:
    单步预测安全盾，保证局部动作不过硬危险。

Expert:
    风险感知局部决策器，主要降低风险与分析 failure modes。

PPO student:
    学习 Expert residual 倾向，在低在线计算成本下减少部分安全干预负担。

do-no-harm gate:
    当上层策略持续无效或减速进入风险墙时，回退到 A*+APAS，避免 harmed case。
```

### 9.8 本阶段最终结论

这一阶段可以收尾。

不是因为所有指标都完美，而是因为系统已经形成了完整、可解释、可诊断、可消融的闭环：

```text
可预测层 A* 全局规划
随机层 true-world 执行
局部风险观测
APAS 安全过滤
waypoint 进度修复
Expert 局部风险决策
PPO student 行为蒸馏
do-no-harm harmed-case rollback
failure trace / targeted seeds / 100 episodes 验证
```

继续推进时，不应再在当前 v2.5 内继续堆模块。后续如果开启新阶段，更合理的方向是：

```text
1. 更严格的 holdout seed 评估；
2. 更规范的 DAgger / offline RL；
3. 更真实的 local risk observer；
4. 更正式的 predictive safety filter；
5. terrain/NFZ-aware local planner 或 replan；
6. 面向论文/复试/答辩的叙事重构。
```

但这些已经属于 v2.6 或后续研究，而不是 v2.5 工程收口必须完成的内容。
