# v2.5 阶段性成果与问题复盘

更新时间：2026-06-22

这份文档用于记录本项目从早期 A* + RL 路线，到 v2.5 双层地图、局部风险观测器、Expert/MPC-like 决策器这一阶段的设计演化、阶段成果、失败原因和后续路线。它不是命令说明，而是给后续重构、答辩、复试/面试、论文叙事使用的技术复盘。

## 1. 一句话概括

本项目从“在已知地图上用 A* 规划，再让 RL 做路径修正”的早期混合规划系统，升级为“可预测层用于全局规划，随机层用于真实执行扰动，局部风险观测器提供有限感知，Expert/RL 在执行过程中进行局部安全修正”的双层真实世界仿真框架。

目前最重要的阶段性结论是：

- A* 在可预测层上规划是稳定的，但它无法处理随机层中的局部风险。
- 直接让 PPO/RL 学会避障非常困难，尤其是在风险出现稀疏、延迟惩罚明显、部分可观测、任务失败类型混杂的情况下。
- 引入局部风险观测器和有限步候选动作评估后，Expert 可以作为一个保守的局部安全增强器。
- 在最新的 core+halo 风险模型中，Expert 在 100 episodes 的 fragile 压力场景下将成功率从 42% 提升到 52%，且 rescued=10、harmed=0。
- 这说明 Expert 对“可观测风险”有效，但对 sensor-blind core 这类观测不到的破坏性核心仍然无效。

## 2. 早期路线：v1.0 的 A* + RL

早期系统的核心结构是：

```text
地图/风场/能耗建模
    ↓
A* 全局路径规划
    ↓
RL 对 A* 路线进行修正或优化
    ↓
仿真执行与能耗/风险统计
```

这个阶段已经有不少工程内容：

- 地形、障碍物、风场、能耗等基础模型。
- A* 路径规划。
- PPO/RL 训练环境。
- 任务执行、结果统计、可视化雏形。
- 多机/维护站/状态机等系统方向的初步探索。

但这个版本存在一个关键问题：RL 的任务定义并不够清楚。

如果 A* 已经在同一个可预测地图上给出较优路径，那么 RL 再对它做修正，意义容易变成“微调能耗或速度”，而不是解决一个真正由信息差带来的规划问题。换句话说，早期版本中 RL 并没有明显掌握一个 A* 无法处理的信息优势。

这也是后来 v2.5 的出发点：让 A* 和 RL/Expert 的分工更合理。

## 3. v2.5 的核心升级：双层地图

v2.5 的基本设想是把真实世界拆成两层：

```text
可预测层 Predictable Layer
    terrain / NFZ / forecast wind / predictable storm
    A* 可以使用

随机真实层 Random True-World Layer
    local wind pulses / destructive storm / hidden risk
    A* 不知道，执行时才通过局部观测器感知
```

系统分工变成：

```text
A*:
    负责可预测层上的全局路径、地形约束、大方向。

Local Risk Observer:
    给飞行器一个有限范围内的局部风险摘要。

RL / Expert:
    根据局部观测对 A* 指令进行残差修正。

Safety/Expert:
    在危险动作出现时做局部避险、兜底和保守修正。
```

这个设计相比 v1.0 更合理，因为它明确制造了一个信息差：

```text
A* 只知道预测世界。
执行器面对真实世界。
局部策略的价值来自实时感知随机层。
```

## 4. 随机层现在包含什么

目前 v2.5 的随机层主要包含两类扰动。

第一类是局部风扰动。

它主要用于制造：

- 实际航迹偏移。
- 能耗变化。
- A* 预测路径与真实执行轨迹之间的差异。
- 局部修正策略的必要性。

第二类是破坏性雷暴。

早期设计中，破坏性雷暴更像“突然命中的核心区”。这会导致一个严重问题：如果风险在被观测到之前没有任何可用预兆，那么任何控制器都很难处理。它不是决策器弱，而是信息结构本身不合理。

因此最新版本把破坏性雷暴改成：

```text
外层风险包络 outer risk envelope
    ↓
中间预警外晕 warning halo
    ↓
中心破坏核心 destructive core
```

对应配置在 `configs/config.py`：

```python
v25_destructive_storm_halo_ratio_range = (0.68, 0.86)
v25_destructive_storm_halo_danger_floor = 0.45
```

这个改动的意义不是“让问题变简单”，而是让破坏性风险符合更合理的感知逻辑：核心区域不应该完全没有外围预兆。这样系统才有机会在进入核心之前发现风险，并执行局部避险。

## 5. 局部风险观测器

当前默认观测器是 `circle_oracle`，更适合在论文或汇报中称为 Local Risk Observer，而不是 radar。它不是现实雷达模型，而是一个有限范围、因果的局部风险摘要器。

当前配置：

```python
v25_sensor_mode = "circle_oracle"
v25_radar_radius_m = 1400.0
v25_circle_oracle_samples = 96
```

它的工作方式大致是：

- 以飞行器当前位置为中心。
- 在局部圆形区域内采样风险。
- 前方扇区采样更密，因为飞行器主要关心前进方向。
- 输出局部最大风险、平均风险、最近危险方向、危险质心方向、局部风信息、前方风险等摘要特征。

需要注意：

- 它不是完整 costmap。
- 它不能保证覆盖所有微小核心。
- 它是局部稀疏摘要，不是真实传感器物理仿真。
- 因此系统还引入了 segment probe/safety checker 对候选运动段做硬安全检查。

更稳妥的叙事是：

```text
Local Risk Observer 提供有限范围内的随机层摘要；
Safety Checker 对候选运动段进行局部硬安全验证；
Expert 根据这些观测与验证结果进行有限步局部决策。
```

## 6. PPO/RL 为什么学不好避障

这次迭代中一个重要发现是：RL 避障不是“把随机层塞进观测里就自然会学会”。

RL 要学会避障，至少需要满足三件事：

```text
看得见危险
奖励能提前区分危险
训练中反复遇到相似危险
```

项目中 PPO/RL 遇到的困难包括：

- 风险惩罚往往在失败时才强烈出现，信用分配困难。
- 破坏性核心命中是稀疏失败，样本效率低。
- A* 本身很可靠，RL 的改动空间小，乱动反而容易伤害成功案例。
- 局部避险需要“绕开后回主路”，而不是单步偏航。
- 部分可观测场景中，单帧观测不一定足够判断风险趋势。
- 奖励函数中成功、能耗、路径偏差、安全之间存在冲突。

最新 100 episodes 中，旧 PPO 在 core+halo 新分布下表现明显退化：

```text
A*:       42%
Expert:   52%
RL:       36%

RL rescued: 1
RL harmed:  7
```

这说明当前旧 PPO 不能代表最终 RL 能力。它是在旧分布上训练的，风险模型改变后，策略退化是正常的。后续如果继续做 RL，更合理的路线不是直接从零盲训，而是先使用可靠 Expert 作为教师策略，再做 imitation learning / RL fine-tuning / residual policy。

## 7. Expert / MPC-like 决策器的定位

当前 Expert 不是严格意义上的完整 MPC，也不是全局规划器。它更准确地说是：

```text
有限候选动作集
    +
短时域 rollout
    +
局部风险评分
    +
硬安全约束
    +
do-no-harm gate
```

它的作用是对 A* 的局部指令做残差修正，而不是重新规划整条路。

当前动作空间包含：

- heading residual
- speed residual
- AGL residual

普通候选动作较温和，emergency 候选动作更激进。配置在 `configs/config.py`：

```python
v25_expert_heading_actions = (-0.65, -0.35, 0.0, 0.35, 0.65)
v25_expert_speed_actions = (-0.25, 0.0, 0.35)
v25_expert_agl_actions = (0.0, 0.45)
v25_expert_rollout_horizon = 3

v25_expert_emergency_heading_actions = (-1.0, -0.75, 0.75, 1.0)
v25_expert_emergency_speed_actions = (-1.0, -0.75, -0.50)
v25_expert_emergency_agl_actions = (0.0, 1.0)
```

Expert 的模式包括：

```text
normal:
    无明显局部风险，基本跟随 A*。

cautious:
    有风险感知，但候选动作相对零动作没有足够收益，因此不轻易干预。

avoiding:
    普通候选动作可以明显降低风险，执行局部避让。

emergency:
    普通候选动作都不安全，进入更强的紧急动作集合。

recovering:
    局部风险较低但偏离路径较大，尝试回归 A* 路径。
```

这套模式解决了一个很关键的问题：早期 Expert 加强感知以后，会变得过度保守，导致 timeout 或 harmed。do-no-harm gate 的思想是：

```text
如果零动作已经安全，而候选动作没有明显降低风险，就不干预。
```

这让 Expert 从“到处乱救”变成了“只在必要时谨慎出手”。

## 8. 信息泄露问题与修正

早期 Expert rollout 有一个潜在问题：在短时域预测时，如果直接查询未来时刻的随机层状态，就会产生 future information leakage。

这会让 Expert 变成“预知未来的控制器”，结果虽然可能变好，但不符合真实在线决策设定。

修正后的逻辑是：

```text
rollout 可以预测自身运动学；
但随机层观测时间冻结在当前已观测时刻；
也就是说 Expert 不能直接读取未来随机层。
```

相关测试包括：

- `test_expert_rollout_freezes_random_layer_time`
- `test_segment_probe_detects_core_between_endpoints`

这个修正很重要，因为它把系统从“跑分技巧”拉回到真实在线决策问题。虽然这会让性能变差，但实验结论更可信。

```

这说明 Expert 的收益不是靠随机乱动得到的。它救了 10 个 A* 失败的案例，并且没有伤害 A* 已经成功的案例。

但代价也很明显：

mission_time_s_mean:

energy_used_j_mean:
    A*:     472549
    Expert: 528589

path_distance_m_mean:
    A*:     3718
    Expert: 4108
```

解释：Expert 是保守安全增强器，不是更短路径规划器。它用时间、能耗和绕行距离换成功率。

这组结果非常关键。

因此当前结论应当克制表述为：

```text

```

## 10. 当前遇到的问题

### 10.1 RL 避障没有自然学会

最初以为只要给 RL 局部风险观测，它就能学会避障。但实际发现这很难。

原因包括：

- 避障是安全约束问题，不适合完全交给 reward。
- 失败信号太稀疏，撞核心才惩罚会太晚。
- 局部绕行需要时序策略：提前偏离、绕开、回归。
- PPO 对这种稀疏高风险、部分可观测、长时程任务的样本效率不高。
- 如果 A* 已经很强，RL 的每一次干预都可能变成负贡献。

当前策略：先构造可靠 Expert，再考虑让 RL 学 Expert 或超过 Expert。

### 10.2 Expert 有效但很窄

Expert 当前不是通用智能体。它像一个局部安全模块，擅长处理一部分可观测风险。

它不擅长：

- 完全观测不到的核心命中。
- 大尺度绕行。
- 绕行后复杂回归。
- 地形/NFZ 导致的全局不可行。
- 需要长期等待或重新规划的场景。

因此不能把它包装成“全局避障算法”，更适合称为：

```text
local risk-aware safety expert
```

或者：

```text
short-horizon local safety controller
```

### 10.3 sensor blind core

这是当前最明确的硬问题。

含义：

```text
飞行器在失败前没有足够早地观测到危险预兆，
或者局部观测摘要没有捕捉到核心风险，
导致 Expert 无法提前做出有效动作。
```

最新 core+halo 后，sensor_blind_core 从之前 8/100 降到 6/100，但仍然存在。

这类问题不是简单调 Expert 权重能解决的。可能需要：

- 增大或自适应调整局部风险观测范围。
- 使用前向 corridor 观测，而不是固定圆形摘要。
- 把飞行速度与观测距离耦合起来：速度越快，需要看得越远。
- 对高风险 halo 做更强的预警膨胀。
- 触发局部重规划或全局重规划。

### 10.4 terrain/NFZ

有些失败来自地形或禁飞区，不是随机层风险本身。

这说明局部避险会带来副作用：绕开风险后，可能偏离 A* 安全走廊，进入地形/NFZ 约束更复杂的区域。

这类问题更像重规划问题，而不是单步控制问题。

可能方向：

```text
当前位置 → 重规划回原 A* 路径
当前位置 → 重规划到目标
当前位置 → 重规划到局部安全走廊
```

### 10.5 gradual 风险

gradual 是逐渐增强的风险。

这类风险理论上最适合历史快照和趋势预测，因为它不是突然出现的。如果局部风险在多帧内持续增强，系统可以提前判断：

```text
风险在接近
风险在扩张
风险方向相对机体发生变化
当前航线即将穿过风险区
```

当前 Expert 已经在 gradual 上取得明显提升：

```text
A*:     45.45%
Expert: 66.67%
```

后续可以进一步加入 history-aware features，例如：

- forward risk delta
- local danger area growth
- danger centroid bearing change
- estimated closing speed
- time-to-risk

## 11. 为什么这件事开始像科研问题

早期项目更像工程拼装：

```text
A* + 风场 + 能耗 + RL + 可视化
```

这些模块各自有意义，但问题定义还不够锋利。

v2.5 之后，问题变成：

```text
在部分可观测随机扰动环境中，
如何结合可解释全局规划、局部风险感知、短时域安全决策和学习型残差策略，
提高无人机路径执行的鲁棒性？
```

这个问题比“用 RL 优化 A*”更像科研，因为它涉及：

- 信息可得性：A* 知道什么，执行器知道什么。
- 部分可观测性：局部观测是否足够。
- 风险建模：什么风险可预测，什么风险不可预测。
- 安全约束：哪些动作必须硬拒绝。
- 学习与控制的分工：RL、Expert、A* 各自负责什么。
- 失败归因：不是只看成功率，而是拆分 failure modes。

换句话说，真正有价值的不是“我把 PPO 接上了”，而是“我发现 PPO 接上并不自动解决问题，于是建立了可观测性、风险类型、局部安全决策和失败分类这一整套分析框架”。

## 12. 面试/复试可以怎么讲

可以用下面这段作为主叙事。

> 我做的是一个无人机路径规划与执行仿真项目。早期版本是 A* 全局规划结合 PPO 残差修正，但后来我发现这个问题定义不够合理，因为 A* 和 RL 面对的是同一张可预测地图，RL 的价值不明显。于是我把环境升级成双层地图：可预测层给 A* 做全局规划，随机真实层在执行时才通过局部观测器暴露给控制策略。这样 A* 负责全局可解释路线，局部策略负责随机扰动下的安全修正。

然后讲遇到的关键困难：

> 一开始我以为 PPO 看到局部风险就能学会避障，但实际效果并不好。后来我分析发现，避障不是简单奖励问题，它涉及部分可观测、稀疏失败、延迟惩罚和安全约束。于是我先构造了一个短时域局部安全 Expert，用候选动作 rollout、segment safety check、hard constraint 和 do-no-harm gate 来做保守避险。

再讲阶段结果：

> 在最新 fragile 压力场景的 100 episodes 中，A* 成功率 42%，A*+Expert 成功率 52%，Expert 相对 A* rescued=10、harmed=0。进一步做 hazard classification 后发现，Expert 对 gradual 和 sudden 这类可观测风险有效，其中 gradual 从 45.45% 提升到 66.67%，sudden 从 47.73% 提升到 56.82%；但对 sensor-blind core 仍然无效。这说明系统不是简单调参变好，而是明确暴露了可观测风险和不可观测风险的边界。

最后讲反思：

> 这个项目让我意识到，学习算法本身不是万能的，关键是问题建模。A*、Expert、RL、Safety Checker 应该有清晰分工。后续我计划把 Expert 作为教师策略，用 imitation learning 或 RL fine-tuning 训练策略，同时对 sensor-blind core 做观测器和重规划机制改进。

## 13. 当前推荐后续路线

### P0：固化当前成果

- 把 core+halo 风险模型作为当前 v2.5 的稳定阶段。
- 保留 100 episodes comparison 和 hazard classification 结果。
- 不急着继续乱调 PPO。
- 当前 Expert 可以称为 stage result，而不是临时补丁。

### P1：分析 sensor_blind_core

目标：弄清楚剩余 6 个 sensor_blind_core 为什么发生。

问题清单：

- 是局部观测半径不够吗？
- 是速度太快，看到时已经来不及了吗？
- 是采样摘要漏掉了小核心吗？
- 是核心生成在 A* 必经路线附近吗？
- 是 halo 预警强度不够吗？

### P2：history-aware gradual predictor

目标：利用历史快照处理 gradual 风险。

候选特征：

- 风险最大值变化率。
- 前向风险变化率。
- 危险面积增长。
- 危险质心方位变化。
- 与飞行方向的相对接近速度。

这可以作为“局部风险趋势预测”，比直接说 MPC 更准确。

### P3：rejoin / replan

目标：解决避障后偏离 A* 路径、terrain/NFZ 或 timeout 的问题。

可能方案：

- 偏离过大时重规划到原路径。
- 偏离过大时重规划到目标。
- 建立局部安全走廊。
- 在 Expert 模式中增加 REPLAN 触发。

### P4：RL 重新定位

不要再把 RL 当成凭空学会避障的主力。

更合理路线：

```text
Expert 产生教师数据
    ↓
行为克隆 / imitation learning
    ↓
PPO/SAC fine-tuning
    ↓
RL 学会在 Expert 基础上减少能耗、缩短时间、改善回归
```

这样 RL 的意义是：

- 学习 Expert 的局部安全反应。
- 在安全边界内优化能耗和效率。
- 逐步超过手写 Expert，而不是从零解决全部避障问题。

## 15. 当前最重要的结论

当前项目不要再用一句“A*+RL 路径规划”概括。更准确的说法是：

```text
一个面向随机扰动真实执行场景的分层无人机路径规划框架：
A* 负责可预测层全局规划，
Local Risk Observer 提供随机层局部感知，
Expert/RL 执行局部残差修正，
Safety Checker 保证候选动作安全，
hazard classification 用于诊断不同失败类型。
```

当前阶段最好的成果不是“成功率终于涨了 10%”，而是：

```text
我们证明了在合理的 core+halo 风险模型下，
局部安全 Expert 对可观测风险确实有效，
同时明确了 sensor-blind core 是下一阶段必须解决的信息问题，
而不是简单的控制器调参问题。
```

这才是这轮 v2.5 迭代真正变硬的地方。
